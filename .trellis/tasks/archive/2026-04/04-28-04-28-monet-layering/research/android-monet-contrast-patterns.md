# Research: Android Monet Contrast Enhancement Patterns

- **Query**: How does Android's own system UI handle Monet surface contrast? Any AOSP patterns worth replicating?
- **Scope**: External (AOSP knowledge, Android 14+ system UI patterns)
- **Date**: 2026-04-28

## Findings

### Android's Monet Implementation

Android 12 (API 31) introduced Monet (Material You dynamic color). The system extracts colors from the wallpaper and generates a `CorePalette` using the same `material_color_utilities` library (ported to Kotlin/Java in AOSP as `packages/apps/SystemUI/src/com/android/systemui/monet/`).

### Android System UI Surface Handling

Android's system settings, Quick Settings, and notification shade use Material 3 surface tokens. Key observations:

1. **Default contrastLevel is 0.0** in AOSP -- the same "flat" look that Flutter apps experience
2. **Android 14 (API 34) introduced user-selectable contrast**: Settings > Accessibility > Color and motion > Color contrast provides three levels:
   - **Default** (contrastLevel = 0.0)
   - **Medium** (contrastLevel = 0.5)
   - **High** (contrastLevel = 1.0)
3. This is the same `ContrastCurve` system used in `material_color_utilities`

### AOSP Contrast Implementation

In AOSP's `frameworks/base/core/java/android/content/res/Configuration.java`, a `contrastLevel` float field was added in Android 14:

```java
// AOSP Configuration.java (Android 14+)
public float contrastLevel;
```

This value is passed to `DynamicScheme` when resolving colors, using the same `ContrastCurve` interpolation that the Flutter `material_color_utilities` package implements.

### Android System UI Patterns for Visual Layering

Android System UI uses several techniques beyond color contrast:

1. **Tonal Elevation + Shadow**: Cards and surfaces use `elevation` to trigger subtle tonal shifts, not shadows. In Material 3, elevated surfaces get a slight tint from `surfaceTint` color blended at low opacity. This is separate from surface container tokens.

2. **Outline borders**: Some system UI elements use `outline` or `outlineVariant` as a 1dp border instead of relying solely on background color differences. This provides visual separation even when surface tones are similar.

3. **State layers / Overlay**: Interactive elements use `stateLayer` overlays (ripple/color overlays at fixed opacities like 8%, 12%, 16%) to indicate state, providing additional visual differentiation.

4. **Icon and text contrast**: System UI ensures text and icons have sufficient contrast against their backgrounds, even when surface-to-surface contrast is minimal. The `onSurface` and `onSurfaceVariant` tokens use `ContrastCurve(4.5, 7, 11, 21)` which provides high contrast even at default level.

### Key Pattern: Subtle Border + Tint Elevation

A pattern worth replicating:

```dart
// Instead of relying only on background color difference:
Container(
  decoration: BoxDecoration(
    color: colorScheme.surfaceContainerLow,
    border: Border.all(
      color: colorScheme.outlineVariant.withValues(alpha: 0.3),
    ),
    borderRadius: BorderRadius.circular(12),
  ),
)
```

This adds a faint outline that provides visual separation even when `surface` and `surfaceContainerLow` are nearly identical.

### Android's Neutral Palette Tonal Range

The neutral palette generated by Android's Monet has extremely low chroma (typically 2-4 in HCT). This means:
- At tone 98 (surface): nearly pure white/light gray
- At tone 96 (surfaceContainerLow at default): nearly identical to surface
- The perceptual difference between tone 98 and 96 is ~1-2% lightness

This is why the "flat" appearance is a fundamental issue with the default Monet palette when using fixed tone values. Android's own solution was to add the `contrastLevel` parameter.

### Relevance to Flutter Implementation

The AOSP pattern maps directly to Flutter:

| Android Pattern | Flutter Equivalent |
|---|---|
| `contrastLevel` in `DynamicScheme` | `ColorScheme.fromSeed(contrastLevel: 0.5)` or manual `DynamicScheme` construction |
| Outline borders | `Border.all(color: colorScheme.outlineVariant)` |
| `surfaceTint` elevation | `Material(surfaceTintColor:)` (enabled by default in M3) |
| State layers | `MaterialStateProperty` / widget state overlays |

## Caveats / Not Found

- Android 14's contrast setting is a system-wide preference, but the `dynamic_color` Flutter plugin does NOT read it. The plugin only retrieves the `CorePalette`, not the user's contrast preference.
- There is no Android API available to Flutter (via the `dynamic_color` plugin or otherwise) to query the user's contrast preference from Settings > Accessibility.
- AOSP source references are from public knowledge of Android 12-14 releases; specific file paths may vary by Android version.
- On non-Android platforms (iOS, macOS, desktop), there is no Monet/CorePalette system at all. The `dynamic_color` plugin falls back to `ColorScheme.fromSeed(accentColor:)` on macOS/Windows/Linux.
